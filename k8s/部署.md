### 启动集群
- 创建kind-3node.yaml文件

ymal
```yml
kind: Cluster                     # 告诉 Kind 这是一个集群配置文件
apiVersion: kind.x-k8s.io/v1alpha4  # 配置文件的 API 版本，固定写法
name: dev                         # 指定集群名称，默认为 "kind"，这里改为 "dev"
nodes:                            # 定义集群中的所有节点
  - role: control-plane           # 第一个节点角色：控制平面（master）
    kubeadmConfigPatches:         # 对 kubeadm 初始化配置进行补丁修改
      - |                         # 表示接下来是一个多行字符串（YAML 的块标量）
        kind: InitConfiguration
        nodeRegistration:
          kubeletExtraArgs:
            node-labels: "ingress-ready=true"
    # ↑ 这段补丁的作用：
    # 在 kubeadm init 阶段， kind: InitConfiguration:给 kubelet 额外参数 node-labels，指定这个补丁对应的 kubeadm 配置对象类型为 InitConfiguration。kubeadm 的配置分为多个阶段：InitConfiguration（初始化阶段）、ClusterConfiguration（集群全局配置）、KubeletConfiguration（kubelet 配置）等。这里我们只修改初始化阶段的配置。
    #nodeRegistration：开始定义节点注册相关的配置（nodeRegistration下的内容会影响 kubelet 在加入集群时的行为，例如设置节点标签、污点、额外参数等）。#kubeletExtraArgs：定义要传递给 kubelet 的额外命令行参数
    # 使该节点拥有标签 ingress-ready=true。给这个控制平面节点添加一个标签 ingress-ready=true，这个标签通常用于 Ingress Controller（如 ingress-nginx）的调度策略。许多 Ingress 控制器会通过 nodeSelector或亲和性规则要求 Pod 只运行在带有 ingress-ready=true标签的节点上。因为我们在控制平面节点上暴露了 80 和 443 端口，所以给它打上此标签，让 Ingress 控制器能正常调度到这里。
    # 许多 Ingress 控制器（如 ingress-nginx）会通过 nodeSelector 只调度到有这个标签的节点。
    extraPortMappings:            # 额外的端口映射：宿主机端口 → 容器端口
      - containerPort: 80         # 容器内端口 80（HTTP）
        hostPort: 80              # 宿主机端口 80
        protocol: TCP
      - containerPort: 443        # 容器内端口 443（HTTPS）
        hostPort: 443             # 宿主机端口 443
        protocol: TCP
    # ↑ 这两条映射让宿主机可以直接通过 localhost:80 和 localhost:443 访问到
    # 控制平面节点内运行的 Ingress Controller 服务（通常监听 80/443）。
    # 注意：端口映射只在控制平面节点上做（因为 Ingress 通常会调度到 control-plane）。
  - role: worker                  # 第二个节点：工作节点
  - role: worker                  # 第三个节点：另一个工作节点
```

### 装 metrics-server（kubectl top 必备）

  kubectl top是 kubectl 的一个子命令，用来实时查看 Kubernetes 集群中节点（Node）或 Pod 的资源使用情况（CPU 和内存）。它类似于 Linux 里的 top命令，但针对的是 Kubernetes 集群

- 部署

powershell
```powershell
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
如果卡住，可以先下载components.yaml，然后执行

#访问下载
https://github.com/kubernetes-sigs/metrics-server/releases

#执行
kubectl apply -f components.yaml

#执行可能失败还是下载镜像的问题
1.查看镜像版本
kubectl describe deployment metrics-server -n kube-system | Select-String "Image:"


2.改镜像，从集群拉取kube-system命名空间下名为 metrics-server的 Deployment 的当前完整配置，然后修改。改已经加载的components.yaml，或者在kubectl apply -f components.yaml执行前修改components.yaml
kubectl edit deployment metrics-server -n kube-system
修改成下面的：
containers:
- args:
  - --cert-dir=/tmp
  - --secure-port=10250
  - --kubelet-preferred-address-types=InternalIP,ExternalIP,Hostname
  - --kubelet-use-node-status-port
  - --metric-resolution=15s
  - --kubelet-insecure-tls        # ← 确保有这一行
  image: registry.cn-hangzhou.aliyuncs.com/google_containers/metrics-server:v0.8.1

或者直接修改components.yaml，然后执行
kubectl delete -f components.yaml
kubectl apply -f components.yaml

如果修改错误使用命令行（莫名奇妙不成功）

kubectl patch deployment metrics-server -n kube-system --type=json `
-p='[{"op":"add","path":"/spec/template/spec/containers/0/args/-","value":"--kubelet-insecure-tls"}]'

#导出
kubectl get deployment metrics-server -n kube-system -o yaml > metrics-server-fix.yaml
#用编辑器打开后修改（同上）
#重新apply
kubectl apply -f metrics-server-fix.yaml
#验证参数
kubectl get deployment metrics-server -n kube-system -o jsonpath='{.spec.template.spec.containers[0].args}'
#重启
kubectl rollout restart deployment metrics-server -n kube-system


#3.查看pod状态
kubectl get pods -n kube-system -l k8s-app=metrics-server -w
    │        │              │                       │
    │        │              │                       └── -w = watch，持续监听变化
    │        │              └── -l = label selector，只显示标签匹配的 Pod
    │        └── -n = namespace，指定命名空间
    └── 获取 Pod 列表

#select-string=grep
```

新建myapp.yaml

yaml
```yaml
apiVersion: apps/v1                    # 指定 Deployment 的 API 版本（稳定版）
kind: Deployment                       # 资源类型：Deployment，用于管理 Pod 副本
metadata:
  name: myapp                          # Deployment 的名称
spec:
  replicas: 2                          # 期望运行 2 个 Pod 副本
  selector:
    matchLabels:
      app: myapp                       # 选择器：管理带有 app=myapp 标签的 Pod。告诉 Deployment “哪些 Pod 归我管”
  template:
    metadata:
      labels:
        app: myapp                     # 为 Pod 模板设置标签 app=myapp，与选择器匹配。给 Deployment 即将创建的 Pod 打上标签，是 Pod 的实际标签
    spec:
      containers:
        - name: myapp                  # 容器名称
          image: myapp:1.0  # 使用的镜像
          imagePullPolicy: IfNotPresent   # 或 Never
          ports:
            - containerPort: 8081      # 声明容器暴露的端口（仅供文档参考，实际映射由 Service 决定）
---
apiVersion: v1                         # Service 的 API 版本
kind: Service                          # 资源类型：Service，提供稳定的网络入口（稳定的访问入口，负载均衡，服务发现，Pod 的“代理”或“负载均衡器”）
metadata:
  name: myapp                          # Service 的名称
spec:
  selector:
    app: myapp                         # 将流量转发到标签 app=myapp 的 Pod
  ports:
    - port: 80                         # Service 在集群内部暴露的端口（其他服务可通过 myapp:80 访问）
      targetPort: 8081                 # 将流量转发到 Pod 的 8080 端口（容器端口）
---
apiVersion: networking.k8s.io/v1       # Ingress 的 API 版本
kind: Ingress                          # 资源类型：Ingress，提供外部 HTTP/HTTPS 路由
metadata:
  name: myapp                          # Ingress 的名称
spec:
  ingressClassName: nginx              # 指定使用 ingress-nginx 控制器处理此 Ingress
  rules:
    - host: myapp.local                # 匹配的域名（HTTP Host 头必须为此值）
      http:
        paths:
          - path: /                    # 匹配所有路径
            pathType: Prefix           # 前缀匹配模式（以 / 开头的路径都匹配）
            backend:
              service:
                name: myapp            # 转发到的 Service 名称
                port:
                  number: 80           # 转发到 Service 的 80 端口
```
